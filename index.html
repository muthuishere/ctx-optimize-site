ctx-optimize — your repo, as a graph, in a second

A deterministic code knowledge graph for coding agents. One static Go binary. No LLM at query time, no database, no server — the graph serves context, your agent supplies the intelligence.

12 languages embedded ~1s to gather 12k files ~100ms queries 0 runtime dependencies

Why

Coding agents burn their context window on grep round-trips and speculative file reads. The single most expensive pattern we measured: the agent finds a symbol, then opens the whole file just to see its signature. ctx-optimize pre-computes the graph once — declarations, calls, imports, docs — and answers from it in milliseconds, with citations, so the agent reads only what it must.

Deterministic by contract

The binary never calls a model, a database, or the network (except the remote you configure). Same input, same graph, byte for byte. Git-diffable output.

Symbol cards

card Store.Merge returns the signature, doc comment, location, callers and callees in one shot — no file read needed to cite or call a symbol.

Honest edges

Every edge carries confidence: EXTRACTED is parsed fact, INFERRED is name-matched. Ambiguous calls are dropped, never guessed.

Wiki on every gather

Each add regenerates a deterministic markdown wiki — index, hub pages, per-file pages — readable by humans and agents alike.

Extensible-first

Drop a script in .ctxoptimize/adapters/ and it runs on every add. Drop a grammar pack in grammars/ and the language works. No forking the tool.

Team sync, no server

remote push / pull to any S3-compatible bucket. Credentials as ${VAR} names in a committable config — values never written, never printed.

Wiki + exports

Markdown wiki on every add; export to GraphML, CSV, DOT, JSON, or a full Obsidian vault.

Methodology — read this before quoting the numbers. graphify was timed on its fastest deterministic path (update --no-cluster: extraction only, no clustering, no LLM). ctx-optimize was timed on add, which does more work per run: extraction, graph build, stale-node pruning, and full wiki regeneration. Query latency is the median of 5 runs, same question, same 2,000-token budget, both CLIs invoked cold from the shell. Node counts are shown for scale, not equivalence — the tools disagree about what counts as a node (ctx-optimize graphs import modules and markdown documents as first-class nodes; graphify includes some file types we skip), which is why ctx-optimize shows ~34% more nodes on flask while the 12k-file totals land within 1%. On gin and on its own source graphify finds slightly more nodes and edges than we do, and we show that as-is. graphify also does things ctx-optimize deliberately does not: LLM-assisted community labeling and semantic clustering. If you want those, use graphify — the comparison here is the deterministic core both tools share.

Design

What's in the binary, and what never will be.

in:   tree-sitter grammars compiled to WASI, hosted by wazero (pure Go, CGO_ENABLED=0)
      one committable .ctxoptimize/ dir per repo: config.json + adapters/
      central store at ~/ctxoptimize/<repo>/ — multi-module, git-diffable NDJSON
      query · card · explain · path · affected · hubs · wiki · export · serve

out:  no LLM API      the host agent answers; we serve context
      no database     files, sorted, atomic writes
      no embeddings   lexical + graph retrieval, deterministic
      no server       a static binary and your filesystem

Lineage

With all due respect to graphify.

ctx-optimize exists because of graphify, and openly builds on its best ideas — the fail-closed batch door, query budgets, the save-result/reflect feedback loop. The central store and the remote push/pull hook design were contributed to graphify by this project's author — issue #1751 and PR #1752, verifiable on GitHub — before this project existed. Where graphify reaches for an LLM — wiki generation, community naming — ctx-optimize stays deterministic and hands the intelligence back to the host agent. Different bet, shared respect.
